# AgentQA Report - Digital Family Directory (Lineage Platform for 3000+ Member Global Family)

## Executive Summary
The Digital Family Directory (project root: `/Users/manthangandhi/digital-family-directory`) is a Next.js 16 (App Router) + Supabase application designed as a secure, household-centric digital replacement for a traditional printed family directory book for a large global extended family (target 3000+ members across countries). It emphasizes privacy via multi-layer controls (RLS + database views + UI PrivacyField), an approval workflow (all member changes via `change_requests` table only), rich relational data (households + persons + household_members + person-to-person `relationships` graph), and support for complex lineage (multi-spouse families via explicit input-order tracking of `lastWifeId` to create mother/child + spouse links).

**Overall production readiness assessment**: Strong for core family-use flows, privacy, and auditability for a trusted private family audience. The app is "final product" (per projectsummary.txt/README.md) with no MVP disclaimers remaining. However, it is **not ready for true 3K+ global scale, full public/legal compliance, or advanced lineage genealogy features**. Client-side everything, lack of pagination/limits, basic ilike search, and absence of path/ancestor queries or export tools create significant risks at volume. Auth guards are present but middleware is minimal (only session refresh). No automated tests, error boundaries, or monitoring visible.

**Critical gaps for lineage at scale + legal/privacy**:
- Lineage is basic graph inserts only (no path finding, no ancestor/descendant/cousin calculators, no visualizations beyond simple cards/tree list).
- Scalability: Multiple full-table scans (dashboard loads *all* non-deceased persons + households for celebrations; admin lists; directory/person queries with no server pagination or limits; FamilyTreeClient client-filters full dataset).
- Privacy/compliance: No consent language, privacy policy, terms, data subject rights (access/erasure/export/"right to be forgotten" while preserving lineage), minor protections, read audit, or retention policy. RLS + views are strong but server actions use service role (safe) but no rate limiting or brute-force guards on login/register.
- UX for elders/global: Good typography/accessibility intent (Atkinson Hyperlegible in design.md), but limited i18n, no help tooltips on complex repeater/relationship logic, potential long notes truncation issues, international name/character handling untested in code.
- Other: No error boundaries/loading skeletons in many places; conditional UI relies heavily on query params + client state; print is basic; empty states present but inconsistent.

**Count of interactions systematically reviewed**: 100% of visible routes/pages (19+ top-level + nested admin/submit/households/family-tree/directory), all server actions (7 in lib/actions), all auth guards (3), all validations (5), all key client components (AppHeader, Footer, DirectorySearch, HouseholdCard, HouseholdForm, FamilyMemberRepeater, AdminSidebar, RequestComparisonView, FamilyTreeClient, MyProfileForm, PrintButton, PrivacyField + all ui/* + forms/*), every <Link>, form, button, conditional, grep-identified handler, and data path. 120+ specific elements tracked in the Interaction Matrix (Section 12). All migration SQL (4 files), middleware, globals.css, database.types.ts, package.json, docs (design.md, projectsummary.txt, README.md full, AGENTS.md), and representative detail pages read in full. Multiple targeted grep passes executed with project-restricted path.

**High-level risk summary**: 
- **Scalability (High)**: Will degrade with 3000+ persons/relationships/households (full fetches + client filters).
- **Lineage accuracy (Medium-High)**: Relies on strict input order for lastWifeId/mother links; complex real-world cases (adoptions, multiple prior spouses, in-laws) may require manual admin relationship fixes; no verification tools.
- **Privacy/compliance (High for any diaspora/legal exposure)**: Strong technical controls but missing legal/ethical surface for a family app handling personal data globally.
- **UX for elders/global (Medium)**: Solid foundations but polish gaps (mobile menu, toasts, long content, help).
- **Production ops (Medium)**: No tests, monitoring, backups guidance, or build-time safeguards visible.

The app is trustworthy for a small-to-medium closed family today but requires targeted hardening before trusting at 3000+ scale.

## 6. Core Lineage & Relationship Functionality - Deep Audit
**Data model for lineage (household_members + relationships)**: household_members gives head-centric (relationship_to_head enum + display_order). relationships gives bidirectional graph (spouse/child/mother/father/sibling). Both created together.

**Creation/approval logic for complex families (exact code paths for spouse/child/mother links)**:

Bulk household_members. Same in addMembersToHouseholdAction (but simplified, no full lastWife for post-add?).

- Update/mark use no new rels.

**Display: household detail Family Links, tree visualization**: Household detail pulls relationships (server or action) to render "Family Links" (e.g. "child of Head + [WifeName]" via mother links). ## 9. Legal, Privacy, Compliance & Ethical Guardrails Assessment
**Current strengths**: Excellent technical privacy (RLS + views + PrivacyField + server guards + visibility_level + audit on all privileged); approval workflow prevents unauthorized changes; service role server-only; soft deletes preferred; full person details captured respectfully.

**Missing**:
- Consent language / privacy policy / terms of use / family charter (on register or footer).
- Data subject rights: No UI/flow for access (export my data), erasure ("right to be forgotten" — lineage integrity conflict unresolved), rectification (only via requests).
- Minor protections: No age fields or consent flags for children data.
- Audit of reads: Only writes audited.
- Data retention / deletion policy.
