# QAAgentAnalysis Report - Digital Family Directory (Live Run + Exhaustive Checks)

**Executive Summary**

Overall health: The app is a sophisticated Next.js 16 (App Router + Turbopack) + Supabase full-stack platform for a private, approval-gated digital family directory/genealogy system targeting large extended families (explicit 3K+ members goal). It demonstrates strong architectural intent around privacy (RLS + DB views + PrivacyField), auditability, status-based conditionals, and complex relational lineage (households + persons + household_members + relationships graph with lastWife/mother link heuristic). Core flows (auth, submit → request → compare/approve with lineage replay, directory/household views switching on approval, repeater forms) are implemented with server actions, Zod validation, and revalidation.

**Critical blockers (production-killing):** 
- Hard TypeScript build failure in `app/directory/print/page.tsx:108:32`: `Property 'id' does not exist on type 'never'.` (culturalPages.map; exact quote from live `npm run build`).
- ~167+ lint problems (304-line lint output captured; dominant categories: `@typescript-eslint/no-explicit-any` (majority, pervasive in actions, admin pages, directory, forms, components), `react/no-unescaped-entities`, unused vars, `react-hooks/set-state-in-effect` (DirectorySearch, ThemeToggle), `@next/next/no-img-element`).
- No pagination/limits on key queries (full households/persons/relationships/audit fetched in directory, family-tree, dashboard celebrations, admin lists). Client-side filtering on entire datasets in FamilyTreeClient + DirectorySearch sync effects. Scalability risk for 3K+.
- Middleware deprecation warning ("middleware" → "proxy"); minimal real auth enforcement (only session refresh; guards are page/action level).

**Edge Cases, Errors & Scalability Findings**

- Empty: EmptyState used (directory no hh, my-requests); table "No members" colspan.
- Large data: Full fetches (tree: households + members + rels; dashboard persons/households + audit + ann; directory persons limit(50) but hh full ilike). Client filter on all. No range/pagination. Risk: slow + memory at 3K+.
- Complex families: lastWife order-dependent (documented in README/projectsummary); wrong order = missing mother links. Tested in sim (multi WIFE + children). In-laws/OTHER/prior spouses need manual rels.
- Auth states: Visitor (public views + masks + no submit); pending (dashboard alerts, redirects from approved pages); approved (full + submit); admin (sidebar + all actions). Guards redirect correctly (curls + code).
- Privacy: Public view hides contacts; approved sees via member view + PrivacyField. RLS functions enforce.
- Errors: Toasts for action fails; special login confirm; action returns {success, error} handled. No boundaries visible. Partial create (continue on person err, soft inactive).
- Validation: Zod server (min1 members, required names); client pre-checks (at least one field). No full form errors UI.
- Print: window.print + @media; culturalPages never type; large data.
- Other: Hydration (login); middleware deprecation; any casts bypass type safety on lineage; raw img; effect setState (potential loops/rerenders); no tests.
- Supabase: Service role in admin actions (correct for privileged); anon in client.
