# Digital Family Directory - Project Summary (Final Product)

## Overview
This is a complete, production-ready web application called "Digital Family Directory". It serves as a secure, respectful digital replacement for a traditional printed family directory book. The app is designed specifically for one extended family, emphasizing privacy, approval workflows, rich relationship data, and usability for non-technical and elderly family members.

**Key Mindset Shift**: Originally scoped as "MVP v1" with explicit limitations (no complex trees, no full Markdown, etc.), this has been fully evolved into a **final product**. All MVP disclaimers, temporary hacks, and "v1" language have been removed from code, comments, UI, and documentation. The README now serves as the complete user handbook.

The app follows a household-centric model (matching how printed books are organized) while supporting complex family structures (multiple spouses, blended families) through proper relationship tracking.

## Tech Stack
- Next.js 16 (App Router) with TypeScript and Tailwind CSS
- Supabase for Auth, PostgreSQL database, Row Level Security (RLS), and real-time capabilities
- Zod for validation + React Hook Form patterns in client forms
- Server Actions for all privileged operations (no direct client writes to live data)
- Clean, reusable UI components (no heavy external UI libraries)

## Core Data Model
The schema is designed for accuracy, privacy, and family complexity:

1. **profiles** (linked to Supabase auth.users)
   - id, full_name, email, mobile_number, role (member/admin), status (pending/approved/rejected/blocked)
   - Auto-created on signup via database trigger.
   - First admin promoted manually.

2. **households**
   - household_code, primary_member_name, primary_person_id
   - native_place, residence_address, business_address, phone/mobile/whatsapp, email, city/state/country, notes
   - visibility_level, status (active/inactive), verified
   - created_by/updated_by + timestamps

3. **persons** (rich individual records)
   - full_name, gender, date_of_birth, date_of_death, is_deceased
   - education, occupation, marital_status, blood_group
   - mobile_number, whatsapp_number, email
   - notes
   - Full details are captured for every person.

4. **household_members**
   - Links person ↔ household with relationship_to_head (SELF, WIFE, HUSBAND, SON, DAUGHTER, DAUGHTER IN LAW, etc.)
   - display_order, is_primary

5. **relationships** (person-to-person graph)
   - person_id ↔ related_person_id with relationship_type (spouse, child, mother, father, sibling, etc.)
   - This is critical for complex families.

6. **change_requests** (approval workflow - never direct live updates from members)
   - request_type (add_household, update_*, mark_deceased, etc.)
   - proposed_data (JSONB), current_data, status (pending/approved/rejected)
   - submitted_by, reviewed_by, admin_notes, rejection_reason

7. **audit_logs**
   - Every admin/privileged action is logged with old_data/new_data.

8. **cultural_pages**
   - title, category, content (Markdown supported), language, visibility_level
   - Rich formatting for history, gotra, rituals, etc.

**Key Relationship Logic (How "Wife of Which Child/Son" is Identified)**:
- Households are built around a primary head (SELF).
- When adding members via the repeater (in creation or "Add Missed Members" in edit):
  - Add wives first, then their children (natural input order).
  - System automatically creates:
    - `relationship_to_head` links.
    - Spouse links to the head.
    - Child links from the head.
    - **Explicit mother links** from the most recent wife (based on addition order) to her children.
- This creates a proper family graph.
- In views (Household Detail + Family Tree), a "Family Links" section surfaces these relationships so users can clearly see branches (e.g., "Son of Head + Wife1" vs "Son of Head + Wife2").

Every person record contains the **full set of additional details** (education, occupation, blood group, notes, contacts, etc.). These are fully visible in context for approved members.

## Authentication & Authorization (Final Product Grade)
- Supabase Auth + custom profiles.
- Registration creates pending profile (with email + mobile).
- Login works only after email confirmation (Supabase layer) + admin approval (app layer).
- Server-side guards:
  - `requireAdmin()` — all /admin routes and actions.
  - `requireApprovedMember()` — private data and submission pages.
- Pending users see clear messages but can still submit requests and view public data.
- Admins have a "Confirm Email" button to bypass Supabase confirmation friction for family use.
- Recommendation: For family use, disable Supabase email confirmation or rely on the admin button.

## Privacy Model (Multi-Layer)
- **Database Views**: public_*_view (very limited), member_*_view (full for approved).
- **RLS Policies**: Public can only see safe fields; approved members see full active data; admins have full access.
- **UI Layer**: PrivacyField component shows "•••• PRIVATE" for visitors/pending.
- Visitors see: primary name, city/state/country, basic DOB/education.
- Approved members see: full addresses, all contacts, occupation, blood group, notes, Family Links.
- Household detail and directory respect these rules server-side.

## Member Experience (Approved Users)
- **Dashboard**: Personalized welcome, status badges, quick links, guidance for pending/approved/email states.
- **Directory**: Powerful search/filter (name, place, relationship, marital status, alive/deceased). Results show households + persons.
- **Household Detail**: Rich table with Name, Relationship (to head), DOB, full additional details (Occupation, Education/Marital, Blood/Notes), Mobile (protected), and Family Links column (from relationships table).
- **Family Tree**: Interactive, generated view from real household + relationship data. Searchable, shows ordered members with inline links. Visual hierarchy with borders for tree-like structure.
- **My Profile**: Prefilled form for self-updates (full details). Submits as audited correction request.
- **Submit Forms** (all go through change_requests only):
  - New Household: Full form + FamilyMemberRepeater (supports complex multi-spouse input order for correct linking).
  - Correction: User-friendly form (no JSON) for any household/person.
  - Mark Deceased: Simple, respectful form with date + notes.
- **My Requests**: Track all submissions with status and rejection reasons.
- **Culture**: Beautifully rendered Markdown content (history, gotra guidance, rituals, etc.).

## Admin Experience
- Centralized layout with sidebar + consistent top header.
- **Users**: List, approve/reject/block, promote to admin, "Confirm Email" button.
- **Households**: Create with full repeater. Edit basic fields + "Add Missed Family Members" section (reuses repeater). Deactivate (soft delete).
- **Persons**: View/edit rich details.
- **Change Requests**: Queue with side-by-side comparison view. Approve applies changes live + full audit. Reject with visible reason.
- **Cultural Pages**: Full CRUD with Markdown editor (immediate visibility based on level).
- **Audit Logs**: Complete, filterable history of every action.
- **Overview**: Quick stats (pending users/requests, active households) + shortcuts.

## Key Technical & Robustness Features
- **No direct live updates from members** — everything routes through change_requests.
- **Approval applies data correctly**: Creates persons + household_members + relationships (including mother links). Updates, mark deceased, etc.
- **Audit on every privileged action**.
- **Soft deletes** preferred for households/persons.
- **Service role key** used only in server actions (never exposed to browser).
- **Server-side protection** on all sensitive routes and actions.
- **Zod validation** on all forms (with safeExtend for refined schemas).
- **Consistent header**: Single AppHeader component (public vs member vs admin states). No more repetitive per-page navs.
- **Error handling**: Toasts, loading states, empty states, proper redirects.
- **Mobile responsive** with clean, premium-neutral design (large enough targets, clear labels, respectful tone).

## Family Tree Specifics
- Generated automatically from approved data (no manual generation step).
- Shows households as family units with members in display_order.
- Displays inline relationships.
- Client-side search/filter by name.
- Visual structure using borders and indentation for hierarchy.
- Benefits directly from the enhanced relationship creation (mother links).

## Known Limitations / Future Considerations (Honest)
- Family tree is functional and visual but tree-based rather than a full interactive graph (no D3/react-flow) — kept simple and accessible.
- My-profile updates still go through correction requests (by design, to maintain "every change approved" principle).
- No notifications (email or in-app) on request status changes (could be added later).
- Person detail pages are currently embedded in household/family views (could be standalone later).
- Assumes one primary household per person (real-world blended families may need additional modeling).

## How to Run (Production-Ready Instructions)
1. Create Supabase project.
2. Copy .env.example → .env.local and fill keys.
3. Run migrations in order: 001_initial_schema.sql, 002_seed_data.sql, 003_add_email_to_profiles.sql (if needed).
4. Register first user → promote to admin + approved via SQL.
5. (Optional) Use Admin > Users "Confirm Email" button.
6. Start creating households (pay attention to member addition order for correct linking).
7. Share with family.

The app is now a complete, documented, consistent digital family archive ready for real use.
